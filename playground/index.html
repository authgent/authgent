The Problem
OAuth Discovery & Standards
RFC 8414

Server Metadata

Auto-discovery endpoint at /.well-known/oauth-authorization-server. Advertises all endpoints, grant types, scopes, DPoP algorithms. MCP auto-discovery compatible.

RFC 7517

JWKS Public Keys

JSON Web Key Set at /.well-known/jwks.json. Contains active + rotated ES256 public keys for JWT verification. SDKs fetch and cache automatically.

RFC 9728

Protected Resource Metadata

At /.well-known/oauth-protected-resource. Lets resource servers advertise which authorization server they use and which scopes they accept.

DPoP — Sender-Constrained Tokens
RFC 9449

DPoP Proof Verification

Clients generate an ephemeral ES256 key pair and attach a signed DPoP proof JWT to every request. The server verifies: typ=dpop+jwt, method (htm), URL (htu), freshness (iat), and access token binding (ath). Tokens become sender-constrained — stolen tokens are useless without the private key.

Stateless

HMAC-Based Nonces

DPoP nonces are computed as HMAC-SHA256(server_secret, time_bucket) — no server-side storage. 5-minute buckets with 1-bucket grace period. The server returns DPoP-Nonce header on 401, client retries with the nonce included. Zero-state, zero-DB-hits.

cnf.jkt

Token Binding

When DPoP is used, the issued token contains a cnf.jkt claim — the JWK Thumbprint (RFC 7638) of the client's public key. Token type becomes DPoP instead of Bearer. Configurable policy: strict (reject non-DPoP), audit (log), or permissive.

Delegation Chain Security
Enforced

Delegation Depth Limit

Configurable max_delegation_depth (default 5). When an exchange would create a chain deeper than the limit, the server returns 403 DelegationDepthExceeded. Prevents infinite agent-to-agent delegation spirals.

may_act

Actor Authorization

Parent tokens can include a may_act.sub claim listing which specific actors are allowed to exchange the token. Any other actor gets 403 MayActViolation. This creates explicit trust chains — the orchestrator decides exactly who can act on its behalf.

Anti-splice

Signed Delegation Receipts

Every token exchange produces a signed JWT receipt stored server-side: token_jti, parent_jti, actor_id, and a SHA-256 chain_hash of the canonical chain JSON. Prevents chain-splicing attacks where adversaries graft delegation chains together.

RFC 8707

Resource Indicators

The audience parameter in token exchange maps to RFC 8707 Resource Indicators. Agents can have allowed_exchange_targets restricting which resources they can delegate to. Matching modes: exact (full URL) or origin (scheme+host).

Pre-flight

Token Exchange Pre-Check

POST /token/check — validates subject token, client, grant type, exchange targets, scope subset, delegation depth, and may_act — all without issuing a token. Lets agents verify permissions before committing to a delegation.

OIDC-A

Agent Identity Claims

Delegation chains propagate agent_type, agent_model, agent_version, agent_provider, and agent_instance_id through every hop. Also preserves human_root — the original human identity that started the chain.

Additional OAuth Grant Types
OAuth 2.1

Authorization Code + PKCE

Full browser-based consent flow with mandatory S256 PKCE. Shows a consent page (or auto-approves in dev mode). Issues access + refresh tokens. CSRF protection with HMAC-SHA256 tokens. The foundation for human-initiated agent authorization.

RFC 8628

Device Authorization

For input-constrained agents (CLI tools, IoT). Agent gets a device code + user code, displays a verification URL. Human approves on a separate device. Agent polls until approved. Human-readable codes with no ambiguous characters (no 0/O/l/1).

Rotation

Refresh Token + Replay Detection

Refresh tokens use family-based tracking. Each use rotates the token and returns a new one. If a previously-used token is replayed (stolen), the entire family is invalidated — all tokens in the lineage become dead. Atomic CAS operations prevent race conditions.

External Identity Provider Bridge
Bridge

Auth0 / Clerk / Okta id_token Exchange

Agents authenticated by an external IdP can exchange their id_token for an authgent access token via token exchange. The server fetches and caches JWKS from trusted issuers, validates signatures, and carries forward idp_iss and idp_sub through the delegation chain. Configure via AUTHGENT_TRUSTED_OIDC_ISSUERS.

Cache

Per-Issuer JWKS Cache

Each external issuer gets an independent JWKS cache with TTL expiry and async lock (thundering-herd protection). On unknown kid, one forced re-fetch is attempted to handle key rotation. Python and TypeScript SDKs have matching JWKSFetcher classes with the same caching behavior.

Audit & Observability
Events

Audit Trail

Every token issuance, exchange, revocation, and replay attempt is recorded. Query via GET /audit with filters (action, actor, client_id, since) and pagination. 90-day retention with automatic cleanup.

Redaction

Structured Logging

structlog with JSON output. Regex-based redaction of 11+ sensitive patterns: client_secret, access_token, password, Bearer/DPoP/Basic tokens. Request IDs (ULID) + W3C traceparent propagation for distributed tracing. Secrets never appear in logs.

Inspect

Token Inspector

GET /tokens/inspect?token=... — decode any JWT without verification. Shows claims, delegation chain tree, DPoP binding, expiry status. Works on expired and revoked tokens. Useful for debugging.

Architecture
Pluggable

7 Provider Interfaces

AttestationProvider (agent attestation), PolicyProvider (authz policy — plug in OPA/Cedar), HITLProvider (webhook delivery with HMAC + retry), KeyProvider (signing key storage), EventEmitter (audit), ClaimEnricher (custom claims), HumanAuthProvider (login). All configurable via AUTHGENT_*_PROVIDER env vars with dotted import paths.

Security

5 Middleware Layers

Input sanitization (null byte rejection before parser), Rate limiting (sliding window, per-path, RFC 9457 429 responses), Request ID (ULID + W3C traceparent), CORS (configurable origins, always allows .well-known), Error handler (RFC 9457 Problem Details for non-OAuth, RFC 6749 for token endpoints, never leaks stack traces).

Crypto

Keys Encrypted at Rest

Signing keys stored as AES-256-GCM ciphertext (nonce_hex:ciphertext_hex). Key Encryption Key (KEK) derived from master secret via HKDF-SHA256 (RFC 5869). Separate subkeys for DPoP nonces, CSRF tokens, sessions, and KEK — all from a single AUTHGENT_SECRET_KEY.

Errors

14 Typed Error Classes

InvalidGrant (400), InvalidClient (401), InsufficientScope (403), ScopeEscalation (403), DelegationDepthExceeded (403), MayActViolation (403), TokenRevoked (401), StepUpRequired (403), InvalidDPoPProof (401), UseDPoPNonce (401), and more. All return RFC 9457 Problem Details.

Cleanup

7 Background Tasks

Automatic cleanup of: token blocklist (1h), authorization codes (15m), device codes (15m), refresh tokens (1h), step-up requests (1m), audit logs (90-day retention, daily), delegation receipts (30-day retention, daily). Zero-maintenance — the server manages its own state lifecycle.

Config

40+ Settings

Pydantic Settings with AUTHGENT_* env prefix. Token TTLs (access, refresh, exchange, auth code), registration policy (open/token/admin), consent mode (ui/headless/auto_approve), DPoP policy (strict/audit/permissive), trusted OIDC issuers, webhook config, resource match mode, and more.

SDKs & Integrations
Python

Python SDK

pip install authgent — verify_token(), verify_delegation_chain(), DPoPClient, JWKSFetcher, AgentAuthClient. Middleware for FastAPI and Flask. Adapters for MCP servers, LangChain, and RFC 9728 Protected Resource Metadata.

TypeScript

TypeScript SDK

npm install authgent — mirrors Python SDK. Uses jose for JWT (works Node + edge + browser). ESM-first with CJS compat, tree-shakeable. Middleware for Express and Hono. MCP adapter + scope challenge handler. 1 runtime dependency.

MCP

Scope Challenge Handler

Auto-detects 403 insufficient_scope from protected MCP endpoints. Automatically initiates HITL step-up, polls for human approval, and retries the original request with the elevated token. Zero-code scope escalation handling for MCP tools.
